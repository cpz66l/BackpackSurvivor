## 一、这节课实现了什么

掉落物从"会消失的奖品"升级为"会投进玩家怀里的货"：

```
Idle（原地自转）→ 玩家进磁吸半径 → Attracted（加速飞向玩家）→ 进拾取半径 → 广播拾取事件 → 回池
```

改动与新增：

| 脚本 | 变化 | 职责 |
|------|------|------|
| `PickUpMagnet`（新增） | BS.GamePlay.Loot | 磁吸状态机：Idle/Attracted 两态，加速移动，拾取判定 |
| `DropItem`（改） | +静态事件 `OnCollected`、+`Collect()` 方法 | 从"自转会回收的球"变成"有身份证、会喊话的货" |
| `PickupLogger`（新增·临时） | BS.GamePlay.Loot | 测试订阅者：打印捡到的物品名与稀有度，第 9 课由背包替换 |
| `BasicEnemyLoot.asset`（改） | 条目补齐 dropPrefab 引用 | 掉落物身份数据落地 |

关键设计：**磁吸逻辑独立成组件**，没有塞进 DropItem——一个组件一个职责，以后金币、经验珠可以复用同一个 PickUpMagnet。

## 二、最终代码

### PickUpMagnet.cs —— 磁吸状态机

```csharp
using BS.GamePlay.Combat;
using UnityEngine;

namespace BS.GamePlay.Loot
{
    public class PickUpMagnet : MonoBehaviour
    {
        [Header("磁吸参数")]
        [SerializeField] private float attractRange = 4f; //磁吸半径
        [SerializeField] private float collectRange = 1f; //拾取半径

        [Header("加速度模式")]
        [SerializeField] private float acceleration = 15f;    // 每秒钟速度增加
        [SerializeField] private float maxSpeed = 20f;

        private float currentSpeed = 0f;

        //引用
        private Transform playerTf;
        private Health playerHealth;
        private DropItem dropItem;
        //状态字段
        private MagnetState magentState = MagnetState.Idle;

        private Vector3 direction;

        private void Awake()
        {
            dropItem = GetComponent<DropItem>();
            GameObject player = GameObject.FindGameObjectWithTag("Player");
            playerTf = player.transform;
            playerHealth = player.GetComponent<Health>();
        }

        private void Update()
        {
            if (playerTf == null) return;
            direction = playerHealth.Position - transform.position;
            direction.y = 0f;
            float sqrDistance = direction.sqrMagnitude;

            switch (magentState)
            {
                case MagnetState.Idle:
                    if (sqrDistance < attractRange * attractRange)
                    {
                        StartAttract();
                    }
                    break;

                case MagnetState.Attracted:
                    MoveTowardsPlayer();

                    if (sqrDistance <= collectRange * collectRange)
                    {
                        dropItem.Collect();
                    }
                    break;
            }
        }

        public void StartAttract() => magentState = MagnetState.Attracted;

        private void MoveTowardsPlayer()
        {
            currentSpeed += acceleration * Time.deltaTime;
            currentSpeed = Mathf.Min(currentSpeed, maxSpeed);
            transform.position += direction.normalized * currentSpeed * Time.deltaTime;
        }

        public void StateReset()
        {
            magentState = MagnetState.Idle;
            currentSpeed = 0f;
        }

        private enum MagnetState
        {
            Idle,
            Attracted,
        }
    }
}
```

### DropItem.cs —— 改动部分（收货口）

```csharp
// 类顶部新增：静态事件——全场景拾取公告板
public static event Action<LootEntry> OnCollected;

// Initialize 参数从 Rarity 升级为完整 LootEntry（身份证）
public void Initialize(LootEntry lootEntry)
{
    this.lootEntry = lootEntry;
    switch (lootEntry.rarity) { /* 变色逻辑不变 */ }
}

// 新增：语义化的"被拾取"，区别于"超时回收"
public void Collect()
{
    OnCollected?.Invoke(lootEntry);   // 喊话，带上自己的身份
    Recycle();                        // 然后回池
}

// 池复用时连带重置磁吸组件
public void OnGetFromPool()
{
    survivalTimer = 0f;
    pum.StateReset();
}
```

### PickupLogger.cs —— 临时听众（第 9 课由背包顶替）

namespace BS.GamePlay.Loot

    public class PickupLogger : MonoBehaviour

        private void OnDisable() => DropItem.OnCollected -= HandleCollected;

        private void HandleCollected(LootEntry entry)

            Debug.Log($"捡到：{entry.dropPrefab?.name}，稀有度 {entry.rarity}");
        }

## 三、周期链路：一次拾取的完整旅程

```
① 掉落物出场（池 Get）→ OnGetFromPool：survivalTimer=0 + PickUpMagnet.StateReset()
② Idle：每帧算与玩家的水平距离（sqrMagnitude，不开方）
③ 距离² < attractRange² → StartAttract() 切状态
④ Attracted：currentSpeed 每帧累加（acceleration × Δt，封顶 maxSpeed）
        朝玩家水平移动（y 压平，防止钻地/飞天）
⑤ 距离² ≤ collectRange² → dropItem.Collect()
⑥ Collect()：OnCollected?.Invoke(lootEntry) 广播 → Recycle() 回池
⑦ 订阅者（今天 PickupLogger，明天背包）收到 LootEntry：物品身份 + 稀有度
```

注意 ⑥ 的顺序：**先广播再回池**——和第 7 课"先掉落再回池"同一条规则，事件结算永远在对象休眠前完成。

## 四、设计思考（这节课真正学的东西）

### 1. 收货口的本质：生产者不认识消费者

掉落物不知道背包存在，只负责广播"我被捡了，我是谁"。今天听众是 Debug.Log，第 9 课换成背包，**掉落物一行代码都不用改**。这就是事件驱动解耦的价值。

### 2. 静态事件 = 全局公告板

`static event` 属于类而非实例：订阅者不用满世界找掉落物引用，一处订阅全场景收听。和 TargetRegistry 是同一思路。代价：**静态事件寿命 = 程序寿命**，订阅者必须 `OnDisable -=` 退订，否则引用被永久拽住 → Unity 内存泄漏经典来源。

### 3. `?.Invoke` 是事件触发的标准姿势

没人订阅时事件字段是 null，直接 Invoke 会 NRE。`OnCollected?.Invoke(...)` = "有人听才喊"。

### 4. Collect() 与 Recycle() 的语义分层

"被玩家捡了"和"超时消失"是两种不同的死法：只有前者该发货。Collect 包装 广播+回收，PickUpMagnet 只调 Collect，不碰 Recycle——调用方不越权。

### 5. 加速磁吸的手感来源

匀速 MoveTowards 像机器人；`速度随时间累加 + 上限封顶` 才有"唰一下进身"的爽感。加速度、maxSpeed、attractRange 三个参数就是手感调校的旋钮。

### 6. StartAttract() 的公共语义

它只做一件事：切状态。以后磁铁道具全屏吸取，就是对场上所有掉落物调一遍它——零改动复用。**公共方法设计时要问：未来的谁会调用它？**

## 五、踩坑记录

| 坑 | 根因 | 教训 |
|----|------|------|
| PickupLogger 报 NRE | 掉落表资产里 4 个条目的 dropPrefab **全是 null**（fileID: 0），日志第一次读它就炸了 | 代码没问题时查数据。NRE 标准打法：双击报错跳堆栈 → 该行每个 `.` 左边的引用都是嫌疑人 |
| `UnityEditor.ShaderGraph...` 的 using 混进运行时代码 | IDE 自动补全乱加 using | **`UnityEditor.*` 命名空间打包后不存在，引用 = Build 必炸**。写完文件扫一眼 using 区 |
| 被吸过的掉落物下次出场直接高速起飞 | StateReset 只重置状态机，忘了 currentSpeed | **池复用 = 所有运行期状态都要归零**，漏一个就是灵异 bug |
| StartAttract 语义跑偏（当成逐帧移动用） | 公共方法和私有方法职责没分开 | 公共入口（切状态）和私有实现（逐帧移动）分开，未来的调用方才接得进来 |
| 文件名/类名拼写 Magent | 建文件时手滑 | **重命名要在 Unity 编辑器里做**，在文件管理器改会丢 .meta 引用 |

## 六、面试问答

**Q：Unity 里 `obj?.name` 有什么坑？**
A：Unity 有"假 null"机制——对象被 Destroy 后引用不是真正的 null，`?.` 不会短路，继续访问抛 MissingReferenceException。规则：纯 C# 对象放心用 `?.`；UnityEngine.Object 且可能被销毁的，用 `== null` 判。

**Q：静态事件有什么风险？**
A：静态事件活到程序退出，订阅者忘了 `-=` 退订，引用就被永久持有 → 内存泄漏。所以订阅必须配生命周期：`OnEnable += / OnDisable -=`。

**Q：距离判断为什么用 sqrMagnitude？**
A：magnitude 内部有开方，sqrMagnitude 没有。每帧跑几十上百次距离比较时，把半径也平方后比平方，省掉开方开销。这是 Unity 性能优化的入门肌肉记忆。

**Q：拾取为什么用事件而不是直接 Find 背包组件调 AddItem？**
A：直接调用 = 掉落物依赖背包，测试时没背包就 NRE，换游戏类型要重写。事件 = 生产者零依赖，谁订阅谁消费，单元测试挂个假订阅者就能验证。小项目直接调也能活，规模一大必返工。

**Q：对象池复用时最容易漏什么？**
A：运行期累积的状态归零——计时器、速度、状态机、颜色。漏一个，对象第二次出场就带"前世记忆"。

## 八、思考题（附复习参考）

**Q1：权重随机为什么用"累加区间 + 一次掷点"，而不是逐条判概率？**（第 8 课开课抽查）

> 参考：逐条判概率，所有条目概率和不一定为 1，可能全不中或中多个，还得额外处理。累加区间把权重铺在数轴上，一个随机点落在谁的区间就选谁——**一次掷骰，必中且只中一个，概率天然归一**。

**Q2：为什么拾取要设计成"掉落物发事件"，而不是掉落物直接 Find 玩家背包调 AddItem()？**

> 参考：直接调用让生产者依赖消费者——背包还没写就没法测，背包改名全项目编译错，掉落系统没法单独单元测试。事件让掉落物只依赖"LootEntry 这个数据结构"，消费者随时可换（Logger→背包→成就系统同时听）。小项目直接调能跑，但系统一多就会缠成蜘蛛网。

**Q3：磁铁道具让 100 个掉落物同一帧到达玩家，100 个事件同一帧触发，可能出什么问题？怎么缓解？**

> 参考：① 订阅者（背包）单帧处理 100 次 AddItem，若其中有 UI 刷新会卡顿——解法是**脏标记+帧末批量刷新**（设个 dirty 标记，LateUpdate 统一刷一次）；② 100 个掉落物同帧回池再被取出，池操作集中在同一帧——实际影响小，但大数量级可用分帧处理（每帧最多处理 N 个）；③ 音效同帧叠 100 次会爆音——实际项目会限制同音效并发数。核心思路：**事件可以逐条发，副作用可以合并处理**。
