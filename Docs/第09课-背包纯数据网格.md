## 一、这节课实现了什么

背包系统的**数据内核**——一个完全不碰 UnityEngine 的纯 C# 网格，回答了背包玩法的四个根本问题：能不能放、放、拿掉、哪有地方放。

| 产物 | 说明 |
|------|------|
| `BS.Inventory.asmdef` | 第一个自定义程序集，勾选 No Engine References，物理隔离 Unity API |
| `Item.cs` | 纯数据类：Id / Width / Height，只读属性，构造函数赋值 |
| `InventoryGrid.cs` | 本课主角：6×8 二维数组 + 五大操作 + OnChanged 事件 |
| `InventoryDeBugTest.cs` | 临时 MonoBehaviour 测试驱动，六条剧本全绿 |

**关键认知**：这节课没有一行代码需要在 Unity 里"跑起来看效果"——全部逻辑用 Debug.Log 剧本验证。这就是纯数据层的样子。

### InventoryGrid.cs

```csharp
using System;

namespace BS.Inventory
{
    public class InventoryGrid
    {
        public event Action OnChanged;

        private readonly int width;
        private readonly int height;
        private readonly Item[,] cells;// 二维数组，注意逗号语法

        public InventoryGrid(int width, int height)
        {
            this.width = width;
            this.height = height;
            cells = new Item[width, height];
            //cells[x, y]，x 在前
        }

        public bool CanPlaceAt(int x, int y, Item item)
        {
            if (item == null) return false;
            if (x < 0 || y < 0 || x + item.Width > width || y + item.Height > height)
                return false;
            for (int i = 0; i < item.Width; i++)
            {
                for (int j = 0; j < item.Height; j++)
                {
                    if (cells[x + i, y + j] != null) return false;
                }
            }
            return true;
        }

        public bool Place(int x, int y, Item item)
        {
            //如果存在同一个实例就不允许再放
            if (Contains(item)) return false;
            //如果这个位置不可以放就返回false
            if (!CanPlaceAt(x, y, item)) return false;
            for (int i = 0; i < item.Width; i++)
            {
                for (int j = 0; j < item.Height; j++)
                {
                    cells[x + i, y + j] = item;
                }
            }
            OnChanged?.Invoke();
            return true;
        }

        public void Remove(Item item)
        {
            if (item == null) return;
            bool removed = false;
            for (int i = 0; i < width; i++)
            {
                for (int j = 0; j < height; j++)
                {
                    if (cells[i, j] == item)
                    {
                        cells[i, j] = null;
                        removed = true;
                    }
                }
            }
            if (removed) OnChanged?.Invoke();   // 没删到东西不发事件
        }

        public bool Contains(Item item)
        {
            for (int i = 0; i < width; i++)
            {
                for (int j = 0; j < height; j++)
                {
                    if (cells[i, j] == item) return true;
                }
            }
            return false;
        }

        public Item GetItemAt(int x, int y)
        {
            if (x < 0 || y < 0 || x >= width || y >= height) return null;
            return cells[x, y];
        }

        public bool TryFindFreeArea(Item item, out int x, out int y)
        {
            for (int j = 0; j < height; j++)
            {
                for (int i = 0; i < width; i++)
                {
                    if (CanPlaceAt(i, j, item))
                    {
                        x = i;
                        y = j;
                        return true;
                    }
                }
            }
            x = -1;
            y = -1;
            return false;
        }
    }
}
```

## 三、一次放置的完整校验链

```
Place(x, y, item)
  │ ① Contains(item)？     → 同一实例已在包中，拒（物理不可能原则）
  │ ② CanPlaceAt：
  │     ├─ item == null？   → 拒
  │     ├─ 锚点矩形出界？    → 拒（x<0 / y<0 / x+W>width / y+H>height，一行全覆盖）
  │     └─ 逐格查 cells[x+i, y+j] 有主？→ 拒
  │ ③ 通过 → 逐格写入引用
  └ ④ OnChanged?.Invoke()  → 数据变了，通知世界（第10课 UI 就来订这个）
```

未来的完整链路预览：

```
怪物死 → 掉落 → 磁吸 → Collect → OnCollected(LootEntry)
      → 【第10课】背包监听 → new Item(...) → TryFindFreeArea → Place → OnChanged → UI 重绘
```

## 四、设计思考（这节课真正学的东西）

### 1. asmdef：给代码分房间

默认所有脚本挤在 `Assembly-CSharp` 一个程序集里，改一个文件全量重编。asmdef 把代码切成独立编译单元：**编译加速、依赖显式化**，勾上 **No Engine References** 后这个房间物理隔绝 UnityEngine——背包内核永远不可能被 Unity API 污染，还能脱离引擎跑单元测试。代价是 `Vector2Int` 这类 Unity 类型不能用了，坐标退回两个 int——这不是妥协，是纯数据层该有的样子。

### 2. 格子里为什么存"物品引用"而不是 bool

bool 只能回答"有没有"，引用能回答"是谁"。GetItemAt O(1) 拿到物品，UI 显示、合并匹配、联动检测全都靠它。存 bool 的话，第 10 课必然返工。

### 3. 询问接口 + 动作接口并存，动作接口内部仍校验

CanPlaceAt（问）和 Place（做）分开，是调用方的便利；Place 内部**再调** CanPlaceAt，是防御性设计——公共入口不信任任何调用方。校验和写入遍历的是同一个矩形，两处逻辑锁死同步，永不漂移。

### 4. 实例 ≠ 种类（本课最重要的概念）

`new Item("gun1")` 两次 = 两个实例、一种物品。class 的 `==` 比的是**引用**。Contains 守卫拦的是"同一实例占两块地"（物理不可能），放行"两把同 Id 手枪共存"——因为 GDD 12.6 的合并升级（2 把 1 级白手枪 → 1 把 2 级）**以同类共存为前提**。第 11 课的合并匹配规则"Id 相同 && Level 相同 && 非同一实例"还会再用一次这个区分。

### 5. ★ O(1) 之争：HashSet 方案的思维全记录

**起因**：Contains 是全扫描 O(n)，有没有 O(1)？

**方案**：`HashSet<Item>`——Place 时 Add、Remove 时 Remove，Contains 变 O(1)，且 HashSet 默认比较器对 class 就是引用相等，语义严丝合缝。

**发现的代价**：引入 HashSet = 引入**冗余状态**——"物品在不在包里"存在 cells 和 HashSet 两处，必须签下同步契约：每一条写入路径都要同时维护两边，否则两个结构各说各话，不报错的静寂 bug。

**最终决策：不用**。6×8 = 48 格的全扫描本就飞快，O(n) 的 n 小到没有体感；而同步契约是未来所有新写入路径（合并、替换、移动）都要背的债。**收益≈0，成本>0，不加。**

**带走的思维**：先问渐近成本，再想冗余同步。以及——面试被问"为什么不用 HashSet 优化"时，这段权衡过程本身就是满分答案。另一个埋下的雷：哪天 Item 重写了 Equals/GetHashCode（按 Id 比较），HashSet 语义会从"实例相等"悄悄变成"内容相等"——冗余方案还有隐性耦合。

### 6. 数据层也配拥有事件

`event Action OnChanged` 是纯 C# 特性，不需要 Unity。数据层广播"我变了"，UI（第 10 课）订阅重绘——**UI 不轮询，数据驱动 UI**。Remove 加了 removed 标记，没实际变化不发事件：广播也要有成本意识。

## 五、踩坑记录

| 坑 | 根因 | 教训 |
|----|------|------|
| CS0122：`item.Width` 不可访问 | Item 字段写成 `private readonly`，封过头了 | 写保护要留，读得放行：`public int Width { get; }` 只读属性，构造函数可赋值，外人只读 |
| 命名大小写拧反 | 把"参数小驼峰"理解成了"属性小驼峰" | 铁律：**公开成员 PascalCase，参数/局部变量 camelCase**。改名用 VS 的 F2 重构，调用点自动全改 |
| 测试假阳性 | 想测 item2 重叠，代码里写的却是 item1，结果 false 是对的、验证的东西是错的 | **测试通过 ≠ 测试有效**。每条剧本都要追问：它绿了，是因为它该绿的原因吗？ |
| 命名空间 `BS.Text` | Test 手滑打成 Text | 命名空间 = 目录路径，Tests 目录 → BS.Tests |
| 同一实例占两块地 | Place 没有"实例唯一"守卫 | 已修：Place 先 Contains。这个 bug 是 review 时从测试笔误里顺藤摸出来的——**小 bug 背后常有大漏洞** |

## 六、面试问答

**Q：网格背包的数据结构怎么设计？**
A：二维数组存物品引用（不是 bool），物品以左上角为锚点、占 宽×高 矩形。核心操作四件套：CanPlaceAt（判界+重叠）、Place（防御校验+写入+变更事件）、Remove（引用全扫清除）、TryFindFreeArea（逐锚点首适应）。

**Q：Contains 怎么做到 O(1)？你做了吗？**
A：HashSet 存引用，Place/Remove 时同步增删。但 6×8 网格全扫描只有 48 次比较，引入冗余状态的同步契约成本大于收益，我选择不加。权衡依据：先问渐近成本的实际体感，再评估冗余状态的维护债务。

**Q：class 的 == 比什么？想按内容比怎么办？**
A：默认比引用（ReferenceEquals）。按内容比要重写 Equals + GetHashCode（两个必须一起重写，否则哈希容器行为错乱）。但注意：重写会改变 HashSet/Dictionary 的语义，牵一发动全身。

**Q：TryFindFreeArea 的 out 参数是什么？**
A：方法用返回值报"找没找到"，out 带出"在哪"。调用方 `out int x` 内联声明。没找到时 out 赋 -1 哨兵值。

**Q：asmdef 有什么好处？**
A：编译加速（只重编改动的程序集及依赖方）、依赖显式化（循环依赖编译期暴露）、No Engine References 可造纯 C# 程序集——逻辑可脱离 Unity 单元测试，也是客户端/服务器共享代码的基础。

**Q：Place 返回 bool 和抛异常怎么选？**
A：放置失败是**预期内的常态**（玩家天天把物品往满格里塞），不是异常。返回 false 让调用方平静处理（UI 显示冲突红框）；异常留给真正意外的失败。

## 八、思考题（附复习参考）

**Q1：为什么格子里存"物品引用"而不是 bool 或 id 字符串？**

> 参考：bool 只答"有没有"，字符串要二次查找才知道"是谁"。存引用让 GetItemAt O(1) 直接拿到完整物品（尺寸、稀有度、等级），UI 显示、合并匹配、邻接联动全建立在这上面。空间上只是一个引用（8 字节），划算。

**Q2：支持塔科夫式物品旋转（R 键 90°），数据结构要改哪里？**

> 参考：**数组一格都不用改**。旋转只是"用交换后的宽高走一遍 CanPlaceAt + Place"。要动的：Item 加一个朝向标记（`bool rotated` 或枚举），UI 绘制时按朝向画。这正是"数据与表现分离"的红利——旋转是表现层的旋转，占格逻辑只认当前的宽高。

**Q3：No Engine References 的代价是什么？什么时候你会主动放弃它？**

> 参考：代价是不能用 UnityEngine 的任何类型（Vector2Int、Color、ScriptableObject 全在墙外），数据要跨墙传递时得做转换层。当逻辑天然绑死引擎对象（比如需要直接操作 Transform 的移动控制器）时，隔离没有意义，就该放弃——asmdef 依然值得建（依赖管理+编译加速），只是不勾 No Engine References。

**Q4（加试）：这节课说"UI 只是投影"，怎么理解？**

> 参考：UI 不保存任何玩法状态，它只是把 InventoryGrid 的当前数据画出来；玩家所有操作（拖拽、合并）都是先改数据、数据发 OnChanged、UI 再重绘。状态单一事实源（Single Source of Truth）在数据层——UI 崩了重开，背包数据毫发无损。
