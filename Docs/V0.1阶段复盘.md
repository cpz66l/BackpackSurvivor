## 三、架构状态：三条设计主线

1. **接口驱动**：IDamageable（伤害契约）→ IPoolable（池化契约）。所有系统间协作通过接口，不通过具体类——伤害管线被危险区、近战、自动武器、主动武器 4 种来源复用，零修改。
2. **事件驱动**：OnDamaged / OnDeath 发布-订阅，Health 不知道任何订阅者的存在；死亡流程（注销注册表→归还池子→未来的掉落）全部以挂钩方式挂载。
3. **生命周期对称**：Awake/Start/OnEnable/OnDisable 各有严格分工（自初始化/跨对象/登记注销），为对象池的休眠-复生语义天然兼容——池化落地时敌人侧仅改 3 行。

## 四、关键认知收获（本阶段最重要的 5 条）

1. **"变化时更新缓存"优于"需要时全量查询"**：危险区列表 → TargetRegistry → 刷怪计数，同一模式三次复用。
2. **帧率解耦是数值设计的底线**：tick 结算、减法计时器、RotateTowards×deltaTime——凡是与时间相关的逻辑一律过 Time.deltaTime。
3. **抽象在重复之后（Rule of Three）**：Zone 基类缓建 → 差异数据化；WeaponBase 在第二把武器出现时提炼——两次验证"先具体后抽象"。
4. **生命周期对称是池化的门票**：OnEnable/OnDisable 的登记习惯在第 3 课看似多余，第 5 课兑现为"零改动接入对象池"。
5. **池化对象的完整周期**：预热→Get(SetPool+重置)→服役→Die→Return(防重+休眠)→复生；弹性扩容对象首次归还自动入籍，池容收敛于最高水位。

## 五、踩坑总录（按类型归类）

| 类型 | 案例 | 对策沉淀 |
|---|---|---|
| 空引用 | 守卫无 return / attacker null / 池引用未绑定 | 守卫必带 return；外部输入必 null 兜底 |
| 帧率耦合 | RotateTowards 固定角速度 / 计时器归零漂移 | 一切时间逻辑乘 deltaTime、计时用减法 |
| 物理判定 | Trigger 吃子弹 / 子弹隧道 / 双碰撞体互推 | 显式 QueryTriggerInteraction；扫掠检测；CC 与 Collider 互斥 |
| IDE 误导 | NUnit / Unity.Mathematics / UnityEngine.Pool 乱导入 | 自动导入人工过目，每周 using 大扫除 |
| 生命周期 | Start 订阅在池化后丢失 | 订阅/登记只放 OnEnable/OnDisable |
| 工程环境 | GBK 编码 / 行尾符混用 | .editorconfig 强制 utf-8-bom |
